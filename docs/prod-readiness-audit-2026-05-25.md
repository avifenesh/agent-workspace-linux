# Prod Readiness Audit

Date: 2026-05-25
Last updated: 2026-05-26

This audit maps the active product goal to current evidence. It is intentionally
stricter than "tests are green": a requirement is marked complete only when the
current repo has direct evidence for the requested behavior.

## Verdict

The MCP is production-ready for the current local Linux/X11-backed runtime path
up to the final human review gate.

The implementation has strong automated evidence for permission ceilings,
clean/default advisory action classification, live control, headless semantics,
intent-aware planning, synthetic app/browser dogfood, visible GitHub Explore
dogfood, the GPUI viewer, and thin Codex Desktop integration. The remaining
release gate is manual: final human diff review before staging/shipping.

## Recent Broad Gate

`scripts/prod_readiness_smoke.sh` passed after the real-grocery dogfood
requirements were wired into `mcp_task_plan`, the embedded Codex conversation
screen was removed, reusable frame capture was added, external evidence import
was hardened, duplicate GPUI viewer launches were fenced by a runtime registry
and idempotent MCP `workspace_open_viewer` reuse,
MCP-opened viewers were made target-bound so they exit after the selected
workspace runtime is removed, and direct stdio MCP viewer lifecycle cleanup was
added. It now also covers `workspace_browser_targets`,
`workspace_browser_snapshot`, `workspace_browser_search_results`, and
`workspace_browser_navigate`, which discover, read, extract structured result
cards from, filter GPU-like cards by visible VRAM, warn on event-recording
failure, and navigate workspace Chrome/Chromium page targets from the
running app's browser profile and loopback DevTools endpoint instead of
attaching to the user's host Chrome bridge or using external browser-control
workarounds.
The registry now keys viewer lifecycle separately, so persistent direct
viewers cannot be reused as MCP-bound monitors. `workspace_list_viewers` and
`workspace_close_viewer` are repo-owned recovery tools for cases where the
compositor/window automation cannot see a GPUI viewer; the close path only
signals registered viewer pids whose command line still matches the registry
entry, and `dry_run=true` previews the target. That gate covered:

## Remaining Release Gates

1. Final human diff review:
   Review the large runtime and sibling Desktop diffs before staging. The tree
   intentionally includes many new files and generated dependency lockfile
   changes, so the final review should check scope, packaging expectations, and
   whether any audit docs should move to release artifacts. The release-gate
   audit accepts this only after a human-created
   `target/release-gate-human-review.json` marker with schema
   `agent-workspace-linux.human_final_diff_review.v1`, status `reviewed`, the
   current combined source identity, the current review-scope identity, and
   meaningful reviewer/notes metadata plus `review_artifacts` entries for the
   generated runtime/Desktop review diffs with matching SHA-256 hashes. The
   review-scope identity hashes both repos'
   status, staged and unstaged diffs, and non-ignored untracked file contents in
   a dirty worktree; in a clean worktree it hashes the current `HEAD` commit
   content. This lets a marker bind to either the reviewed dirty diff before
   staging or the reviewed final commit before shipping, and prevents reuse
   after diff, commit, or review-artifact drift. Start from
   the newest `target/final-review-bundle/*.md` bundle;
   it records the combined runtime/Desktop source identity, review-scope
   identity, dirty scope, latest evidence, release-audit/current-source and
   review-scope consistency, pending gates, next evidence commands, checklist,
   generated runtime/Desktop review diffs, and marker template without
   approving anything automatically. When creating the marker after review, set
   `HUMAN_REVIEW_NOTES` to specific non-placeholder notes and pass
   `--notes "$HUMAN_REVIEW_NOTES"`.
   Regenerate the audit and final-review bundle after final doc edits and
   before creating the marker, because docs are intentionally part of the
   review-scope hash.

## Completion Decision

All machine-verifiable requirements in the latest broad gate pass for the
current source hash. Do not mark a production release complete until a human
final diff review marker exists for the current source and review-scope hashes.
Use `scripts/objective_completion_audit.py --require-complete` only after that
review marker is created; until then its normal `pending` status is the
authoritative completion answer.
